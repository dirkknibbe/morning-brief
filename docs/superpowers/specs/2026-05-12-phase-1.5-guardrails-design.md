# Phase 1.5: Guardrails — Design

## Philosophy

> The LLM does creative work inside a sandbox. The sandbox itself is deterministic code.

Every place an LLM agent could screw up gets a non-LLM gate. Bugs in the agent should bounce off a wall, not cascade through the system. This phase ships those walls *before* the factory in Phase 1b puts pressure on them.

## Why now, before Phase 1b

The factory will:
- write arbitrary code,
- run arbitrary bash,
- push branches to GitHub,
- iterate without supervision for an unbounded number of rounds.

Without guardrails, a single bug in the factory loop could push to `main`, hold the build lock forever, run `npm publish`, or burn through a usage window in minutes. We want each of those failure modes blocked at the *boundary*, not inside the agent's reasoning.

## Scope: Tier 1 + Tier 2 only

Tier 1 hardens the existing pipeline. Tier 2 builds the factory's safety primitives so they're available — and tested in isolation — before the factory loop wraps them.

Tier 3 (audit dashboards, telegram heartbeats, stuck alerts) is deferred to after Phase 1b is producing real signals to observe.

## Tier 1 — Defense in depth on existing pipeline

### 1. State-machine validation in `setStatus`

**File:** `src/ideas-state.ts`

Replace the current "any status from `VALID_STATUSES` is fine" check with explicit transition rules:

```ts
const ALLOWED_TRANSITIONS: Record<Status, Status[]> = {
  extracted:   ["queued", "rejected", "needs_human"],
  queued:      ["building", "extracted", "rejected", "parked"],  // can be re-queued
  building:    ["built", "parked", "needs_human", "queued"],     // last resort: re-queue if aborted
  built:       [],                                               // terminal
  parked:      ["queued", "rejected"],                           // can be revived
  rejected:    [],                                               // terminal
  needs_human: ["queued", "rejected"],                           // human can reroute or kill
};
```

`setStatus(db, slug, newStatus, reason?)` reads the current status, checks the transition is in the allowed list, throws a typed `IllegalTransitionError` otherwise. The error includes the current status, attempted status, and the allowed set, so callers and humans both get useful messages.

Tests cover: every legal transition succeeds, a representative sample of illegal transitions fails with the right error.

### 2. Mongo `$jsonSchema` validator on the `ideas` collection

**File:** `scripts/init-db.ts`

Apply a `$jsonSchema` validator to the `ideas` collection. The schema enforces:

- **Always required:** `slug, content_hash, title, raw_text, sources, signal_strength, status, kind, synthesis_depth, created_at, updated_at`.
- **Conditional (via `oneOf` per `status` and `kind`):**
  - `status: building` ⇒ `success_criteria` must be a non-null, non-empty array.
  - `kind: synthesis` ⇒ `parents` must be a non-empty array, `synthesis_thesis` must be a non-empty string, `synthesis_depth >= 1`.

(There is no `built ⇒ factory_run_id` conditional in this phase. `factory_runs` documents reference the idea via `idea_slug`, not the reverse — so the join is already possible without a back-pointer field on `ideas`. Adding the back-pointer would be duplication; if it turns out to matter in Phase 1b for query convenience, the schema can grow then.)

Application:

- For a fresh collection, set `validator` in the `createCollection` call.
- For the existing collection (already populated in Phase 1a), use `db.command({ collMod: "ideas", validator, validationLevel: "moderate", validationAction: "warn" })`. Start in `warn` mode — log violations but accept the write.
- After one full daily cycle confirms zero violations on real data, flip `validationAction` to `error` via a separate one-shot script. This is a deliberate two-step rollout — we don't want a schema mismatch to silently break the daily extract.

### 3. `audit_log` collection + `recordTransition` helper

**Files:** `src/audit.ts` (new), `scripts/init-db.ts` (extend)

New collection `audit_log` with documents shaped:

```ts
{
  slug: string,
  from: Status | null,    // null on first insert
  to: Status,
  actor: string,           // e.g., "extract-ideas", "listener:/reject", "factory:built", "user-cli"
  reason: string | null,
  ts: Date,
}
```

Indexes: `{ slug: 1, ts: -1 }` (per-idea history), `{ ts: -1 }` (recent global activity).

`src/audit.ts` exports:

```ts
export async function recordTransition(
  db: Db,
  slug: string,
  from: Status | null,
  to: Status,
  actor: string,
  reason?: string,
): Promise<void>
```

Hook points:
- `setStatus` calls `recordTransition` after every successful update.
- `applyUpsertOp` (insert path) calls `recordTransition(db, slug, null, "extracted", "extract-ideas")` after a successful insert. The `reinforce` and `skip` paths don't write audit rows — those aren't transitions, they're idempotent re-observations.

The audit log is append-only by convention. No code path deletes from it.

### 4. `system_state` collection + `src/system-state.ts` CLI

**Files:** `src/system-state.ts` (new), `scripts/init-db.ts` (extend)

Single-document collection `system_state`. Document shape:

```ts
{
  _id: "singleton",
  frozen: boolean,                    // true = nothing runs
  extract_enabled: boolean,
  synthesize_enabled: boolean,
  triage_enabled: boolean,
  factory_enabled: boolean,
  freeze_reason: string | null,       // shown in /system-status
  updated_at: Date,
  updated_by: string,                 // "user-cli", "listener:/freeze", etc.
}
```

`init-db.ts` upserts the document on first run with all flags `true` and `frozen: false`. Re-running init-db never overwrites existing flag values.

`src/system-state.ts` exports a library API:

```ts
export type Stage = "extract" | "synthesize" | "triage" | "factory";
export async function isFrozen(db: Db): Promise<boolean>;
export async function isEnabled(db: Db, stage: Stage): Promise<boolean>;
export async function setFrozen(db: Db, frozen: boolean, reason: string | null, actor: string): Promise<void>;
export async function setEnabled(db: Db, stage: Stage, enabled: boolean, actor: string): Promise<void>;
export async function getState(db: Db): Promise<SystemState>;
```

CLI modes:
- `bun run system-state status` → prints frozen + per-stage flags as JSON.
- `bun run system-state check <stage>` → exit 0 if stage may run (not frozen, stage enabled), exit 1 otherwise. Designed for shell gating without a JSON parser dependency. Mongo unreachable → exit 1 (fail closed).
- `bun run system-state freeze [reason]` → sets `frozen: true`.
- `bun run system-state unfreeze` → sets `frozen: false`.
- `bun run system-state enable <stage>` / `disable <stage>` → toggles a per-stage flag.

Add to `package.json` scripts: `"system-state": "bun run src/system-state.ts"`.

### 5. Bash-level kill switch in `loop-triggers.sh`

**File:** `scripts/loop-triggers.sh`

Each fire step is wrapped with a `bun run system-state check <stage>` gate (item 4 added the `check` subcommand precisely so this script doesn't need a JSON parser):

```bash
if bun run system-state check brief 2>/dev/null; then
  fire "triggers/scheduled-brief.md" "brief"
else
  log "skipping brief: frozen or disabled"
fi
```

Same for `action-research`, `extract-ideas`. The `frozen` master switch trumps per-stage flags (the `check` subcommand encodes that logic). Mongo unreachable → `check` exits non-zero → that stage is skipped with a log line. (We *don't* fall through to running anyway when state is unknown — fail closed.)

### 6. Startup flag check in `extract-ideas.ts`

**File:** `src/extract-ideas.ts`

Before the parse phase, after Mongo connects, check `isFrozen(db)` and `isEnabled(db, "extract")`. If either fails the gate, log a one-line reason to stdout and exit 0. (Exit 0 — not 1 — because being frozen is not an error.)

Defense in depth: even if `loop-triggers.sh` is bypassed (e.g., manually invoked), the script self-defends.

### 7. Listener commands `/freeze`, `/unfreeze`, `/system-status`

**File:** `triggers/listener.md`

Extend the "Ideas pipeline commands" section with three new entries that map to `bun run system-state`. Specifically:

- `/freeze [reason]` → `bun run system-state freeze "<reason>"`. Confirm with `🚨 system frozen — all stages will skip until /unfreeze`.
- `/unfreeze` → `bun run system-state unfreeze`. Confirm with `✓ system unfrozen`.
- `/system-status` → `bun run system-state status`. Reply with frozen/enabled flags formatted for Telegram.

The agent must surface these prominently — a frozen system is high-stakes state and needs to be obvious.

## Tier 2 — Factory prerequisites (utilities the factory will use)

### 8. `src/factory-lock.ts` with TTL takeover

**Files:** `src/factory-lock.ts` (new), `src/__tests__/factory-lock.test.ts` (new)

Single-doc collection `factory_lock` (shape from the original spec). Lock document:

```ts
{
  _id: "singleton",
  idea_slug: string,
  started_at: Date,
  ttl_ms: number,
  pid: number,
}
```

API:

```ts
export interface LockResult {
  acquired: boolean;
  prior_owner: { idea_slug: string; started_at: Date; pid: number } | null;
  takeover: boolean;        // true if we acquired by stealing a stale lock
}
export async function acquireLock(db: Db, ideaSlug: string, ttlMs: number, pid?: number): Promise<LockResult>;
export async function releaseLock(db: Db, ideaSlug: string): Promise<void>;
export async function checkLock(db: Db): Promise<LockState | null>;
```

`acquireLock` semantics — two-phase atomic acquire (each phase atomic individually, the standard Mongo lock pattern):

1. **Try insert:** `insertOne({ _id: "singleton", idea_slug, started_at: now, ttl_ms, pid })`.
   - If it succeeds → `{ acquired: true, prior_owner: null, takeover: false }`. Done.
   - If it errors with `code: 11000` (duplicate key) → a doc already exists; proceed to phase 2.

2. **Try takeover-if-stale:** `findOneAndUpdate({ _id: "singleton", started_at: { $lt: new Date(now - <existing>.ttl_ms) } }, { $set: { idea_slug, started_at: now, ttl_ms, pid } }, { returnDocument: "before" })`.
   - First need to read the existing doc to get its `ttl_ms` for the cutoff calculation. Read-then-CAS is *not* truly atomic across both ops, but the CAS in step 2 (filter on `started_at: { $lt: cutoff }`) ensures we only steal if the lock is still stale at the time of update. Worst case: two parallel acquirers both compute the same cutoff, one succeeds the CAS, the other gets a null result and retries phase 1 (which will now error 11000 on the new lock — falling through to phase 2 again with a fresh non-stale lock that cannot be stolen). Bounded race; no double-acquire possible.
   - If `findOneAndUpdate` returns the prior doc → `{ acquired: true, prior_owner: <prior>, takeover: true }`.
   - If it returns null → lock is fresh; `{ acquired: false, prior_owner: <existing-from-phase-2-read>, takeover: false }`.

`releaseLock` deletes the doc only if `idea_slug` matches the caller's. (Defensive: prevents one factory run from releasing a different run's lock if takeover happened.)

Tests cover: acquire when free, acquire when locked (blocks), acquire when stale (takeover succeeds), release matches slug, release with mismatched slug is a no-op.

`mongodb-memory-server` is not added as a dep. Tests use the real Mongo connection (gated on `MONGODB_URI`) and skip cleanly when env is missing — same pattern as the rest of the repo's I/O layer.

### 9. `src/factory-guard.ts` worktree boundary assertion

**Files:** `src/factory-guard.ts` (new), `src/__tests__/factory-guard.test.ts` (new)

```ts
export function expectedFactoryWorktree(ideaSlug: string, repoRoot: string): string;
export function assertInFactoryWorktree(ideaSlug: string, repoRoot: string, cwd?: string): void;
```

`assertInFactoryWorktree` throws `WrongWorktreeError` if `cwd` (defaults to `process.cwd()`) doesn't resolve to `<repoRoot>/.claude/worktrees/factory/<slug>`. Both paths are resolved with `realpath` to handle symlinks.

The factory trigger calls `assertInFactoryWorktree(slug, repoRoot)` immediately after acquiring the lock. If it throws, the factory releases the lock and exits.

Tests are pure: pass synthetic paths via the `cwd`/`repoRoot` args.

### 10. Git pre-push hook + installer

**Files:** `scripts/hooks/pre-push` (new), `scripts/install-hooks.sh` (new), README update.

`scripts/hooks/pre-push` reads stdin (one line per ref being pushed: `<local-ref> <local-sha> <remote-ref> <remote-sha>`) and rejects:

- Push targeting `refs/heads/main`. The user pushes via PR, not directly.
- Push of any tag (`refs/tags/*`).
- Force push (`<remote-sha>` is non-zero AND not an ancestor of `<local-sha>`).

The hook prints a one-line reason to stderr and exits 1.

`scripts/install-hooks.sh` symlinks `scripts/hooks/pre-push` to `.git/hooks/pre-push` (or `.git/worktrees/<name>/hooks/pre-push` if invoked from inside a worktree — git uses per-worktree hooks). Idempotent: re-running just confirms the symlink points at the right file.

README update mentions the install step in Setup. Manually invoked once per clone.

The hook is intentionally simple. Two failure modes we explicitly do *not* try to handle:
- Push from a sibling worktree of `.claude/worktrees/factory/<slug>` — the hook treats those as legitimate factory pushes.
- Bypass via `git push --no-verify` — accepted as a known escape hatch for humans. Agents are blocked from `--no-verify` via the deny list (item 11).

### 11. `.claude/settings.json` deny list

**File:** `.claude/settings.json` (modify)

Add to `permissions.deny`:

```json
{
  "permissions": {
    "deny": [
      "Bash(npm publish*)",
      "Bash(yarn publish*)",
      "Bash(pnpm publish*)",
      "Bash(bun publish*)",
      "Bash(vercel*)",
      "Bash(npx vercel*)",
      "Bash(bunx vercel*)",
      "Bash(gh release create*)",
      "Bash(gh release delete*)",
      "Bash(gh release upload*)",
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(git push --no-verify*)"
    ]
  }
}
```

These bind hook-level. Any agent (factory, listener, action-research) attempting these calls fails immediately with a permission error visible in the transcript.

The deny list is minimal — focused on irreversible/external actions. Local file edits, worktree git ops, and tool installation remain allowed because the factory legitimately needs them.

### 12. Backstop policy declaration

This item is *spec text only*; the implementation lives in Phase 1b's factory loop because the backstops need to wrap the loop.

The factory MUST enforce three independent backstops, each implemented deterministically (no LLM in the termination decision):

- **Stuck detection** (primary, from the original spec): N consecutive rounds with no failing-test reduction AND semantically-similar hypothesis text.
- **Turn ceiling** (secondary): hard cap on total iteration rounds. Default 50 — generous enough to never fire under normal stuck-detection behavior, low enough to bound a runaway.
- **Wall-clock ceiling** (backstop-of-backstop): hard cap on elapsed time. Default 6 hours. Catches the case where stuck detection misfires *and* iterations are slow enough that turn ceiling hasn't fired.

Any backstop firing must:
- Release the factory lock.
- Write `learnings.md` to the worktree describing why it terminated.
- Send a Telegram message.
- Exit non-zero so the wrapping process (tmux session) records the failure.

Phase 1b's factory plan implements these by wrapping the iteration loop in a `BackstopBudget` helper that checks each ceiling per round and raises a typed exception on breach.

## Implementation phasing

**Phase 1.5a (Tier 1)** — items 1-7. Ships immediately, makes the existing pipeline more trustworthy. Required to pass before 1.5b begins.

**Phase 1.5b (Tier 2)** — items 8-12. Ships after 1.5a is merged. The two halves are independent enough to run as separate plans + PRs without coordination overhead.

Each sub-phase is a single plan written via `superpowers:writing-plans`.

## Migration / rollout notes

- **`$jsonSchema` warn → strict**: the validator ships in `warn` mode in 1.5a. After one full daily cycle confirms zero violations against real data, run a one-shot script `scripts/promote-validators.sh` that flips it to `strict`. The script is committed and idempotent. This is *not* automated — Dirk runs it manually after observing the warn-mode logs.
- **Existing 18 ideas in the collection**: all currently have `status: "extracted"` (or whatever they were left as), and the audit_log is empty. Backfill is *not* part of this phase — the audit log starts from "now" and reflects only future transitions. If we want a backfill row per existing idea, that's a one-line follow-up.
- **Pre-push hook installation**: requires manual `bash scripts/install-hooks.sh` per clone. The hook is not magically active. Phase 1.5b's plan includes this install step in its post-merge checklist.

## Open behaviors deferred to operational learning

These are intentionally not pre-tuned:

- **Factory-lock TTL value**: Phase 1b's factory plan picks an initial value based on the wall-clock backstop. Likely "wall-clock backstop + 10 minutes" so a hung factory's lock expires only after the wall-clock backstop has already fired and exited.
- **Turn ceiling and wall-clock ceiling exact values** (50 turns / 6 hours are starting points).
- **Whether `audit_log` needs a TTL index** for size management. Not in v1; revisit if it grows past ~100k rows.

## Out of scope for Phase 1.5

- Telegram heartbeats during builds (Tier 3).
- Daily summary reports / dashboard (Tier 3).
- Mongo collection-level role separation via separate connection strings (the deny-list at the agent layer is sufficient for v1).
- Backstop *implementation* (deferred to Phase 1b's factory loop).
- Backfilling audit_log with one row per existing idea.
- Watchdog process for stuck factory detection — replaced by the lock TTL takeover (passive expiry, no separate process needed).
